# Release v0.0.6

## Goal

Publish a new XReadAgent release that includes the completed PDF upload and BabelDOC
layout-preserving translation fixes. The release should follow the existing tag-driven GitHub
Actions workflow and keep package/app version metadata consistent.

## What I already know

* User requested updating the release and publishing a new version.
* Remote tags were refreshed with `git fetch --tags origin`.
* Latest remote release tag is `v0.0.5`; the next patch release is `v0.0.6`.
* `pyproject.toml`, `frontend/package.json`, and `electron/package.json` currently declare
  `0.0.5`.
* `uv.lock` carries the editable `xreadagent` package version and must be committed with the
  `pyproject.toml` bump because release CI runs `uv sync --frozen`.
* `backend/src/xreadagent/__init__.py` still declares `__version__ = "0.0.2"` and should be
  aligned with the package metadata for source-level consistency.
* `.github/workflows/release.yml` triggers on `v*` tags, builds Windows, and publishes a GitHub
  Release with `softprops/action-gh-release`.
* macOS packaging remains disabled in the release workflow due to the single-arch Python bundle
  contract documented in the Electron spec.

## Requirements

* Bump XReadAgent release metadata from `0.0.5` to `0.0.6`.
* Commit the `uv.lock` editable package version update generated by the metadata bump.
* Align backend source `__version__` with the release version.
* Keep the existing Release workflow publishing model unchanged:
  tag push triggers CI, Windows installer is uploaded, GitHub Release is created by the release job.
* Commit the version bump and Trellis task artifacts.
* Push `main` and tag `v0.0.6` to `origin` to trigger publication.
* Verify the pushed tag or release workflow status when credentials/tools allow it.

## Definition of Done

* Tests/lint/type-check chosen for release metadata risk pass or any skipped checks are explained.
* Release tag is pushed, or a concrete external blocker is reported.
* Trellis task is archived after publish attempt is complete.
* Session progress is recorded in the developer journal.

## Technical Approach

Use the existing patch release convention: bump metadata to `0.0.6`, commit on `main`, create tag
`v0.0.6` at the release commit, then push `main` and the tag. The workflow itself should not be
changed unless inspection reveals a release-blocking defect.

## Decision (ADR-lite)

**Context**: The project already has tag-driven releases and the latest remote tag is `v0.0.5`.

**Decision**: Publish `v0.0.6` as a patch release from current `main`.

**Consequences**: The release includes all commits currently ahead of `origin/main`, including the
PDF source-path/BabelDOC translation fix. Publication depends on GitHub Actions completing the
Windows-only release workflow.

## Out of Scope

* Re-enabling macOS packaging.
* Changing installer signing, auto-update, or release artifact policy.
* Upgrading BabelDOC or other dependencies.
* Creating release notes by hand beyond GitHub's generated notes.
