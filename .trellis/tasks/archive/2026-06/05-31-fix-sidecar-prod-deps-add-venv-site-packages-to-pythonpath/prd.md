# fix sidecar prod deps: add venv site-packages to PYTHONPATH

## Goal

Packaged (production) XReadAgent fails to launch with
`Sidecar exited before becoming ready (code=1)`. Root cause: the bundled Python
sidecar cannot import its dependencies (`ModuleNotFoundError: No module named 'pydantic'`).
Fix the sidecar launch so the bundled base interpreter can find the dependencies
installed in the bundled venv, restoring production startup. Dev mode is unaffected.

## What I already know (diagnosis — reproduced & proven)

- Error surfaced at `electron/src/sidecar.ts:316` (`waitForReady` → process `exit`),
  generic message only. Real error captured by replaying the exact production spawn
  against the installed bundle at `G:\software\XReadAgent`:
  `ModuleNotFoundError: No module named 'pydantic'` (in `xreadagent/agents/crystallize.py`).
- **Bug ①** `electron/src/sidecar.ts:238-251` (`spawnProcess`) sets `VIRTUAL_ENV` +
  prepends venv `Scripts` to `PATH`, and sets `PYTHONPATH = backendPath` only. But the
  base interpreter (`resources/python/python.exe`) **does not read `VIRTUAL_ENV`** to
  resolve `sys.path` — so the venv's `site-packages` (pydantic/fastapi/uvicorn/…) is never
  searched. Verified: base interp's `site-packages` has none of the deps; venv's has all.
- **Bug ②** venv is non-relocatable: `python-venv/pyvenv.cfg` has
  `home = D:\a\XReadAgent\XReadAgent\electron\resources\python` (CI build-machine path).
  So "just launch the venv's `python.exe`" also fails — verified:
  `No Python at 'D:\a\XReadAgent\...python.exe'`.
- **Fix proven** against the live installed bundle: launching the base interpreter with
  `PYTHONPATH = <backend>;<venv>\Lib\site-packages` prints `SIDECAR_READY port=...` and runs
  healthy.
- Spec gap: `.trellis/spec/electron/index.md:99-100` codifies an incomplete contract
  ("production sidecar startup relies on `PYTHONPATH=resources/backend`") — omits venv
  site-packages. Must be updated to prevent regression.
- Build flow (`electron/scripts/bundle-python.mjs`): downloads base python →
  `uv venv` → `uv pip install <root>` into venv → copies backend source. Unchanged by the
  chosen fix.

## Decision (ADR-lite)

**Context**: Bundled sidecar can't import deps in production. Three fix layers possible:
runtime PYTHONPATH (sidecar.ts), build-time install-into-base (bundle-python.mjs), or
relocatable venv.
**Decision**: **Runtime fix in `sidecar.ts`** — add the venv's `site-packages` to
`PYTHONPATH` alongside `backendPath`. Chosen by user. Rationale: empirically proven against
the existing installed bundle, smallest diff, no Python re-bundle required, no change to the
build pipeline.
**Consequences**: Base interpreter + venv site-packages are mixed; safe because both come
from the same CPython 3.12.8 build (matching ABI for compiled ext like `pydantic_core`).
The misleading `VIRTUAL_ENV` mechanism is replaced/clarified. Existing installed app still
needs a rebuild+reinstall (or a separate hot-patch) to benefit.

## Requirements

- In `SidecarManager.spawnProcess()` (`electron/src/sidecar.ts`), when `venvPath` is set,
  compute the venv `site-packages` directory and include it in `PYTHONPATH`, joined with
  `backendPath` via `path.delimiter`, preserving any pre-existing `PYTHONPATH`.
  - Windows: `<venvPath>/Lib/site-packages`
  - POSIX: `<venvPath>/lib/python3.*/site-packages` (resolve actual dir; project pins 3.12)
- Keep behavior identical in dev mode (no `venvPath` → `PYTHONPATH` unset, as today).
- Extract the env-construction into a small pure/testable helper so it can be unit-tested
  without spawning a real process.
- Update `.trellis/spec/electron/index.md` sidecar contract to state production startup
  relies on `PYTHONPATH = resources/backend + resources/python-venv/<site-packages>` (and
  note `VIRTUAL_ENV` alone is insufficient for the base interpreter).
- Add/extend a unit test asserting the constructed env includes the venv site-packages on
  `PYTHONPATH` (mock `child_process`; no real Python — per spec test rules).
