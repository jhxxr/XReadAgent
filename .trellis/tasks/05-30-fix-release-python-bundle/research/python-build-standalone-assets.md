# python-build-standalone Assets For Release Bundle

## Query

Why did Release workflow run `26645229773` fail during `cd electron && pnpm pack:python`?

## Findings

The public GitHub Actions job API shows:

* `Build (Windows)` job `78528836517` failed in step `Bundle Python`.
* `Build (macOS)` job `78528836584` also failed in step `Bundle Python`.
* Both jobs ran from head SHA `6e467e2a00c537312d7738fe9b4694c4660254da`.

The full logs endpoint requires repository admin rights, so stderr could not be retrieved
from the public API. The step name still points directly at `electron/scripts/bundle-python.mjs`.

The script currently targets python-build-standalone release tag `20241219` and Python
`3.12.8`. GitHub's release asset API for `astral-sh/python-build-standalone` shows the
relevant existing install-only assets are:

* `cpython-3.12.8+20241219-x86_64-pc-windows-msvc-install_only.tar.gz`
* `cpython-3.12.8+20241219-aarch64-apple-darwin-install_only.tar.gz`
* `cpython-3.12.8+20241219-x86_64-apple-darwin-install_only.tar.gz`

The current Windows script mapping instead generates:

* `cpython-3.12.8+20241219-windows-x86_64-install_only.zip`

That URL returned HTTP `404`. The real Windows x64 tarball URL returned HTTP `200` after
GitHub's normal release-asset redirect.

## Impact

The Windows release bundle step is guaranteed to fail before the installer build because it
downloads a non-existent archive and then attempts to extract it as a zip. The script should
use python-build-standalone's target-triple style platform suffixes and `.tar.gz` for
Windows.

The macOS asset names generated by the current script match existing assets for both x64 and
arm64. Because macOS also failed in the same step and full logs are unavailable, verification
should cover both generated URL metadata and the tarball extraction path.

After downloading the macOS x64 and Windows x64 tarballs locally, `tar -tzf` showed their
top-level paths start directly with `python/...`, for example:

* macOS: `python/bin/python3`
* Windows: `python/python.exe` plus `python/DLLs/...`

The prior script comment expected `cpython-...-install_only/python/...` and used
`--strip-components=2` for tarballs. With the actual top-level `python/` directory, that
would strip both `python/` and the next component (`bin/` on macOS), leaving
`resources/python/python3` instead of `resources/python/bin/python3`. That explains why
macOS can fail even though its generated URL is valid.
